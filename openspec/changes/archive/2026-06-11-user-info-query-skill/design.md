## Context

当前项目基于spring-ai-alibaba框架构建智能体系统，已实现两个skill（pdf处理和前端设计）。系统通过FileSystemSkillRegistry加载skills目录下的技能。现需要新增用户信息查询技能，支持拼音模糊匹配功能。

## Goals / Non-Goals

**Goals:**
- 实现用户信息查询技能，可通过中文姓名查询用户个人信息
- 支持拼音模糊匹配，处理谐音和输入不准确的情况
- 提供1000个模拟客户信息数据，存储在临时文件中
- 支持通过对话框输出客户信息列表

**Non-Goals:**
- 不实现用户信息的增删改功能
- 不实现持久化存储（使用内存数据）
- 不实现复杂的用户认证和权限控制

## Decisions

### 1. 拼音匹配方案选择

**选择**: 使用pinyin4j库实现拼音转换和匹配

**理由**:
- 已完成性能测试验证，满足项目需求
- pinyin4j成熟稳定，社区支持良好
- 支持多音字处理和拼音首字母提取
- 项目已有相关使用经验（PinyinBenchMark）

**替代方案**:
- pinyin-plus：功能更丰富但增加额外依赖
- 自定义实现：开发成本高，维护困难

### 2. 数据存储方案

**选择**: 临时文件存储 + 内存缓存

**理由**:
- 1000条数据存储在临时文件中，便于维护和更新
- 封装接口读取文件，支持数据热更新
- 启动时加载到内存缓存，保证查询性能
- 文件格式可选择JSON或CSV，便于生成和解析

**实现方式**:
- 数据文件：`data/customers.json`（项目根目录下的data目录）
- 数据加载器：`CustomerDataLoader`负责读取文件并加载到内存
- 缓存结构：`Map<String, UserInfo>`按姓名索引，`List<UserInfo>`全量数据

### 3. 技能集成方式

**选择**: 通过FileSystemSkillRegistry注册SKILL.md文件

**理由**:
- 符合现有项目架构
- 利用框架的技能发现和加载机制
- 便于维护和扩展

## Risks / Trade-offs

**[风险] 拼音匹配准确性** → 通过多音字词库和模糊匹配算法缓解，允许一定误差

**[风险] 性能问题** → 1000条数据通过内存缓存保证性能；文件读取仅在启动时执行一次

**[权衡] 功能简化** → 当前版本仅支持查询，不支持修改；可后续扩展

**[权衡] 数据模拟** → 使用随机生成数据，可能与真实数据格式有差异
